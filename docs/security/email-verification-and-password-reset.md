# Email Verification and Password Reset

Verification and reset links are generated from environment variables:

- `FRONTEND_EMAIL_VERIFY_URL` (tenant users)
- `FRONTEND_PASSWORD_RESET_URL` (tenant users)
- `FRONTEND_PLATFORM_ADMIN_PASSWORD_RESET_URL` (platform admins -- completely separate URL/flow
  from tenant-user reset, never mixed)

Link shape:

```text
{FRONTEND_URL}?token=<raw_token>
```

The tenant is **not** part of the link. Tenant-user reset/verification tokens are resolved
across all tenants by `token_hmac` alone (`consume_password_reset_token_global` /
`consume_email_verification_token_global`), the same cross-tenant lookup pattern used for
login, since the caller does not know (and must not need to supply) which tenant issued the
token. Platform-admin tokens need no such lookup: `platform.platform_users.email` is globally
unique, so platform-admin reset/verification is a direct, single-realm operation against
`platform.platform_password_reset_tokens` / `platform.platform_email_verification_tokens`.

Raw tokens appear only in the email body, generated by `generate_opaque_token()` (32 bytes of
`secrets.token_urlsafe`). The database stores only `sha256:<hmac>` values, computed with
`IDENTITY_PASSWORD_RESET_TOKEN_SECRET` / `IDENTITY_EMAIL_TOKEN_SECRET`. API responses are
enumeration-safe (`"If an account exists, ... instructions have been sent."`) regardless of
whether the email matched zero, one, or more than one active account.

Token consumption is atomic and single-use: `consume_password_reset_token`/
`consume_email_verification_token` (and their cross-tenant `_global` variants) are a single
`UPDATE ... WHERE status = 'pending' AND expires_at > now() RETURNING ...` statement, so a
concurrent or repeated attempt to use the same token fails safely (only one caller can ever see
`status = 'pending'` transition, and reuse always fails).

Reset password revokes all of the user's existing sessions (`revoke_user_sessions`) and sends a
`password_changed_email` notification. Change-password verifies the current password first, then
does the same. Both write a security/audit event (`tenant.security_events` for tenant users,
`platform.audit_logs` with `platform_admin.*` action keys for platform admins).
